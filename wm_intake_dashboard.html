Intake checklist

Click any item to cycle its status: Not started → Done → In progress → Blocked. Items marked Blocked automatically surface in the weekly update.

Access & Setup
Workspace access confirmed
WM team has logged in to the client ESP workspace. Confirm role level — Admin, Support, or Author. Flag if less than Admin.
Owner: WM
Not started
Admin / Support role obtained
Full Admin or Support role is required to see Data In sources, integrations, and pipeline settings. If WM only has Author, raise as a blocker immediately.
Owner: Client
Not started
NDA and project brief signed
Confirm NDA is countersigned and project brief / SOW is agreed before any sensitive data is accessed or shared.
Owner: WM PM
Not started
Kickoff meeting completed
Kickoff with all stakeholders held. Notes documented. Owners confirmed for each phase. Blockers surfaced and logged.
Owner: WM PM
Not started
Data Audit
Profile count and breakdown pulled
Total, active, inactive, with/without email. Any unusual identifier patterns (ID prefixes, phone-only profiles). Billable profile count noted for cost impact.
Owner: WM
Not started
All campaigns inventoried
Full list: name, status, trigger type, delivered volume, open and click rates. Flag any with missing geography tags, broken triggers, or unusually low engagement.
Owner: WM
Not started
Full attribute index audited
Pull every attribute. Identify duplicates (e.g. phone vs phone_number vs phoneNumber), stale fields, non-standard naming conventions. Note which % of profiles each appears on and whether any campaign or segment uses it.
Owner: WM
Not started
All events inventoried and source-typed
List every event. Record source type (iOS SDK / JS / server-side / unknown). Note campaigns and segments that use each. Flag events with X icon (unknown source), stale test events, and any events missing a source field in the payload.
Owner: WM
Not started
Data In sources audited
For each source: confirm type, last received timestamp, attribute names sent, whether dual-destination forwarding is enabled, and whether sample payloads have been obtained. Requires Admin access — raise as blocker if not available.
Owner: WM (requires Admin)
Not started
All workflows and automations documented
List every active workflow/journey. Who built it? Is it healthy? Does it have dependencies on other campaigns, custom objects, or webhooks? Flag any that need to be paused before migration.
Owner: WM + Client
Not started
Sample event payloads obtained from client
Request actual JSON payload examples for the top 5–10 events. Verify attribute names match what's expected, confirm source field presence, and check timestamp format (Unix epoch vs ISO 8601).
Owner: Client Engineering
Not started
Client Decisions Required
Geography logic confirmed
How exactly is US vs International (or other geo splits) determined? Phone prefix? Payment address? IP at registration? Registration country? This must be agreed before any segmentation or tagging work begins.
Owner: Client — decision needed
Not started
All campaigns tagged by geography / audience type
Every active campaign must be labelled (e.g. US / International / Both, or relevant segments) before any migration or interim segmentation work starts. Mis-tagging during this phase is the primary risk to live sends.
Owner: Client + WM to validate
Not started
Migration direction agreed internally
Client has confirmed the migration approach (e.g. single vs split workspace, which profiles move first, timeline constraints) with all internal stakeholders before WM begins migration prep.
Owner: Client — internal sign-off
Not started
SDK / backend naming updates confirmed
Client engineering team has confirmed they will update all attribute names sent via SDK and backend to match the agreed standard (e.g. phone not phone_number). Timeline agreed. This is typically a hard dependency for WM's cleanup work.
Owner: Client Engineering
Not started
Consent / subscription conflict handling agreed
If merging or migrating profiles, what happens when two profiles have different opt-in statuses? Client must decide the rule (more conservative always wins, or active subscription wins) before any merge is executed.
Owner: Client — decision needed
Not started
Governance & Sign-offs
Sign-off process agreed for irreversible actions
Any irreversible action (profile deletions, merges, permanent attribute nulling) must have a documented sign-off process. Who approves? In what format? What review period? Agree this before Phase 1 executes — not at the point of execution.
Owner: WM PM + Client lead
Not started
Rollback / recovery plan documented
For every phase, document what can and cannot be undone. Share with the client. Ensure they understand which actions are irreversible before they sign off on them. This protects WM and sets proper expectations.
Owner: WM
Not started
Migration timing blackout dates confirmed
Identify peak revenue periods (product launches, sports events, sales, holidays) where no migration or risky work should occur. Get these dates from the client and block them out in the project plan before finalising any timeline.
Owner: Client + WM PM
Not started
Weekly status update cadence agreed
Agree with the client how and when WM will communicate progress — weekly email, shared doc, Slack, or scheduled call. Set expectations early so the client is never left wondering what's happening.
Owner: WM PM
Not started
